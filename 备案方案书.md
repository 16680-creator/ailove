## 三、网站服务内容介绍

### 3.1 网站概述

"爱在朝夕"是一款面向情侣的生活记录服务平台，通过微信小程序和 Web 端为情侣提供日常生活记录、照片管理、心愿清单等功能，帮助情侣记录美好时光、增进感情交流。

### 3.2 网站栏目及内容介绍

| 栏目名称 | 内容描述 | 用途说明 |
|----------|----------|----------|
| **首页** | 恋爱计时器、每日情话、快捷入口展示 | 展示情侣基本信息和常用功能入口 |
| **情侣日记** | 日记撰写、心情记录、天气标记、图片附件 | 记录情侣日常生活点滴 |
| **生活相册** | 相册创建、照片上传、瀑布流展示、收藏功能 | 存储和展示情侣照片 |
| **今天吃什么** | 菜品库管理、分类筛选、随机推荐 | 解决情侣用餐选择困难 |
| **心愿清单** | 愿望添加、状态追踪、优先级管理 | 记录和追踪共同心愿 |
| **姨妈助手** | 生理期记录、周期预测、提醒功能 | 女性健康管理 |
| **个人中心** | 用户信息、情侣绑定、设置管理 | 账户管理 |

### 3.3 网站用途

本网站为个人非经营性网站，主要用于：
1. 为情侣提供私密的生活记录空间
2. 记录恋爱历程中的美好瞬间
3. 提供实用的生活辅助工具（菜品推荐、心愿管理等）

### 3.4 域名使用说明

本域名仅用于"爱在朝夕"情侣生活记录服务，不涉及以下内容：
- 不含企业、单位等非个人网站信息
- 不涉及互联网信息服务管理"九不准"等违法违规内容
- 不进行域名交易行为
- 不提供第三方服务或广告推广

## 五、网站安全与信息安全管理制度

### 5.1 网络安全防御措施

#### 5.1.1 网络层安全
| 安全措施 | 实施方案 | 说明 |
|----------|----------|------|
| **防火墙** | 云服务器安全组 | 仅开放必要端口(80/443/22) |
| **DDoS防护** | 云服务商基础防护 | 防止分布式拒绝服务攻击 |
| **入侵检测** | 云盾/安骑士 | 实时监控异常访问 |
| **IP黑名单** | Nginx配置 | 封禁恶意IP地址 |

#### 5.1.2 应用层安全
| 安全措施 | 实施方案 | 说明 |
|----------|----------|------|
| **HTTPS加密** | SSL/TLS证书 | 全站HTTPS，数据传输加密 |
| **CORS策略** | Spring Boot配置 | 限制跨域请求来源 |
| **XSS防护** | 输入过滤+输出编码 | 防止跨站脚本攻击 |
| **CSRF防护** | Token验证 | 防止跨站请求伪造 |
| **SQL注入防护** | MyBatis-Plus参数化查询 | 防止SQL注入攻击 |
| **文件上传安全** | 类型检查+大小限制+病毒扫描 | 防止恶意文件上传 |

#### 5.1.3 数据层安全
| 安全措施 | 实施方案 | 说明 |
|----------|----------|------|
| **数据加密** | AES-256加密敏感字段 | 用户隐私数据加密存储 |
| **密码加密** | BCrypt哈希 | 密码不可逆加密 |
| **数据备份** | 每日自动备份 | 数据库定期备份 |
| **访问控制** | 最小权限原则 | 数据库账户权限控制 |

### 5.2 信息安全管控制度

#### 5.2.1 用户信息保护
1. **信息收集最小化**：仅收集必要的用户信息（昵称、头像）
2. **信息使用透明化**：明确告知用户信息用途
3. **信息存储安全化**：敏感信息加密存储
4. **信息访问严格化**：严格的权限控制和审计

#### 5.2.2 内容管理制度
1. **内容审核机制**：
   - 用户发布内容需符合国家法律法规
   - 禁止发布违法违规、色情低俗、暴力恐怖等内容
   - 建立内容举报机制

2. **内容过滤措施**：
   - 关键词过滤系统
   - 图片内容审核（可接入第三方审核API）
   - 人工审核机制

3. **违规处理流程**：
   ```
   用户举报 → 系统初审 → 人工复审 → 处理决定 → 通知用户
   ```

#### 5.2.3 数据管理制度
1. **数据分类分级**：
   - 公开数据：网站公告、帮助文档
   - 内部数据：系统配置、日志记录
   - 敏感数据：用户个人信息、私密日记
   - 核心数据：用户密码、支付信息（如有）

2. **数据访问控制**：
   - 基于角色的访问控制（RBAC）
   - 操作日志审计
   - 敏感操作二次验证

3. **数据生命周期管理**：
   - 数据采集：明确告知并获得用户同意
   - 数据使用：严格按照收集目的使用
   - 数据存储：加密存储，定期清理过期数据
   - 数据销毁：用户注销后彻底删除数据

#### 5.3.2 应急响应级别
| 级别 | 定义 | 响应时间 | 处理措施 |
|------|------|----------|----------|
| **一级（紧急）** | 系统瘫痪、数据泄露 | 15分钟内 | 立即启动应急预案，通知相关方 |
| **二级（严重）** | 功能异常、性能严重下降 | 30分钟内 | 组织技术团队排查修复 |
| **三级（一般）** | 局部功能异常 | 2小时内 | 记录问题，安排修复 |
| **四级（轻微）** | 体验问题、建议反馈 | 24小时内 | 评估后纳入迭代计划 |

#### 5.3.3 应急处理措施

**1. 数据泄露应急处理**
```
发现泄露 → 立即隔离受影响系统 → 评估泄露范围 → 通知受影响用户
    ↓
报告主管部门 → 配合调查取证 → 修复安全漏洞 → 恢复正常服务
    ↓
总结报告 → 完善安全措施
```

**2. 系统攻击应急处理**
```
发现攻击 → 启动防火墙/封禁IP → 分析攻击来源和方式
    ↓
评估损失 → 修复漏洞 → 恢复服务 → 加固防护
    ↓
保存证据 → 必要时报警处理
```

**3. 服务中断应急处理**
```
服务中断 → 检查服务器状态 → 检查应用日志 → 检查数据库状态
    ↓
定位问题原因 → 尝试快速恢复 → 如无法恢复启用备份方案
    ↓
通知用户 → 修复根本原因 → 恢复正常服务 → 编写事故报告
```
